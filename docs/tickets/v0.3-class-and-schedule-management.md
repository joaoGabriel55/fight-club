# v0.3 — Class & Schedule Management

## Main Goal

Teachers can create, view, update, and delete their classes along with weekly schedules. Bouncer authorization ensures teachers can only manage their own classes. After this ticket a logged-in teacher has a fully functional class management UI.

---

## Scope

### Backend

**Migrations (new)**

- `classes` — `id` (UUID), `teacher_id` (FK → users), `name`, `martial_art`, `has_belt_system` (bool, default false), `description` (nullable), `created_at`, `updated_at`
- `class_schedules` — `id` (UUID), `class_id` (FK → classes), `day_of_week` (0–6), `start_time` (time), `end_time` (time)

**Models**

- `Class` model: `belongsTo` User (teacher), `hasMany` ClassSchedule, `hasMany` Enrollment, `hasMany` Invitation, `hasMany` Announcement
- `ClassSchedule` model: `belongsTo` Class

**Bouncer policy — `ClassPolicy`**

- `view(user, class)` — user is the teacher (class.teacher_id === user.id)
- `update(user, class)` — same
- `delete(user, class)` — same
- `manageStudents(user, class)` — same

**VineJS validators**

- `CreateClassValidator`: `name` (min 3), `martial_art` (min 1), `has_belt_system` (bool), `description` (optional string), `schedules` (array, min 1 entry, each with `day_of_week` 0–6, `start_time`, `end_time`)
- `UpdateClassValidator`: all fields optional
- `CreateScheduleValidator`: `day_of_week`, `start_time`, `end_time`

**Controllers**

`ClassesController` (routes under `/api/v1/classes`, auth middleware on all):

- `POST /` — create class + schedules in a transaction, return `201` with class + schedules
- `GET /` — list teacher's own classes only (teacher_id filter), include schedule count and enrollment count
- `GET /:id` — class detail with schedules (Bouncer: must own class)
- `PUT /:id` — update class metadata (Bouncer: must own class)
- `DELETE /:id` — soft delete class (set `deleted_at`), return `204`
- `GET /:id/students` — list enrolled students: return only `{ id, first_name, belt_level (latest), enrolled_at }` — no email, no birth_date (Bouncer: must own class)

`SchedulesController` (routes under `/api/v1/classes/:classId/schedules`):

- `POST /` — add schedule to class
- `PUT /:id` — update schedule
- `DELETE /:id` — remove schedule

**Serialization**

- Student list response strips all P1/P2 fields. Only `first_name` + latest belt name + `enrolled_at` visible to teacher.

**Audit logging**

- Log `class_created`, `class_updated`, `class_deleted` events.

**Factories & Seeders**

- `ClassFactory` — generates realistic class data
- `ClassScheduleFactory`
- Seeder: creates 2 teacher users each with 2 classes and schedules

### Frontend

**Classes domain**

```
src/domains/classes/
  schemas/
    class.schema.ts       # Zod: createClassSchema, updateClassSchema
    schedule.schema.ts    # Zod: scheduleSchema
  services/
    classes.service.ts    # getClasses(), getClass(), createClass(), updateClass(), deleteClass(), getClassStudents()
  hooks/
    useClasses.ts         # useQuery(['classes'])
    useClass.ts           # useQuery(['classes', id])
    useCreateClass.ts     # useMutation, invalidates ['classes']
    useUpdateClass.ts     # useMutation, invalidates ['classes', id]
    useDeleteClass.ts     # useMutation, invalidates ['classes']
    useClassStudents.ts   # useQuery(['classes', id, 'students'])
  types/
    class.types.ts        # Class, ClassSchedule, ClassStudent types
  components/
    ClassCard.tsx         # Card with class name, martial art, student count, schedule summary, quick actions
    ClassForm.tsx         # RHF + Zod. Multi-section: info fields + ScheduleManager
    ScheduleManager.tsx   # Dynamic field array (useFieldArray): add/remove schedule rows
    StudentList.tsx       # Table of enrolled students with minimal data
```

**Routes**

- `/classes` — grid of `ClassCard` components, empty state if none, "Create class" button
- `/classes/new` — `ClassForm` in create mode
- `/classes/$classId` — Class detail layout (tabs shell; content filled in later tickets)
- `/classes/$classId/students` — `StudentList`
- `/classes/$classId/schedules` — schedule management view (add/edit/remove)

**Auth guard**

- All `/classes/*` routes are under `_authenticated.tsx` layout — redirect to `/login` if not authenticated.
- Teacher-only guard: if `user.profile_type === 'student'`, redirect away from teacher-only routes.

## Acceptance Criteria

- [ ] `POST /api/v1/classes` by a teacher → `201` with class + schedules. The response does not include `teacher_id` (privacy: don't expose internal IDs in responses unnecessarily — use the authenticated user's context).
- [ ] `GET /api/v1/classes` returns only the authenticated teacher's classes — not classes owned by other teachers.
- [ ] `GET /api/v1/classes/:id` by a teacher who does NOT own the class → `403`.
- [ ] `PUT /api/v1/classes/:id` by a non-owner → `403`.
- [ ] `DELETE /api/v1/classes/:id` sets `deleted_at` (soft delete). The class does not appear in `GET /classes` after deletion.
- [ ] `GET /api/v1/classes/:id/students` returns `first_name`, latest belt, `enrolled_at` — no `email`, `last_name`, `birth_date`, `weight_kg`.
- [ ] A student cannot call any `ClassesController` endpoint (profile_type guard → `403`).
- [ ] Schedule `day_of_week` values outside 0–6 → `422`.
- [ ] `end_time` earlier than `start_time` on the same schedule → `422`.
- [ ] Creating a class with no schedules → `422`.
- [ ] Frontend `/classes` page shows a class card for each of the teacher's classes.
- [ ] Frontend create form validates: empty name → inline error, no schedules → inline error. Cannot submit invalid form.
- [ ] After creating a class, the list page updates without a full page reload (TanStack Query invalidation).
- [ ] Deleting a class removes it from the list immediately (optimistic update or invalidation).

## Tests

### Backend (Japa)

- **Create class — happy path:** `POST /classes` with valid payload → `201`, class + schedules in DB, `teacher_id` matches authenticated user.
- **Create class — wrong role:** Student token → `403`.
- **Create class — validation:** No schedules → `422`. Invalid `day_of_week` → `422`. `end_time` < `start_time` → `422`.
- **List isolation:** Teacher A creates 2 classes, Teacher B creates 1. Teacher A's `GET /classes` returns exactly 2. Teacher B's returns exactly 1.
- **Get class — not owner:** Teacher A cannot `GET /classes/:idOfTeacherBClass` → `403`.
- **Update class — not owner:** → `403`.
- **Delete class — soft delete:** After `DELETE /classes/:id`, the row still exists in DB with `deleted_at` set. `GET /classes` no longer returns it.
- **Student list — data stripping:** `GET /classes/:id/students` response objects do not contain `email`, `last_name`, `birth_date`, `weight_kg`, `height_cm`.
- **Schedule CRUD:** Add schedule → appears in `GET /classes/:id`. Update `day_of_week` → reflected. Delete → gone.
- **Audit log:** `class_created` event written after successful creation.

### Frontend (Vitest)

- **ClassCard:** Renders class name, martial art, schedule count. "Delete" button triggers confirmation before calling mutation.
- **ClassForm — schedule field array:** Clicking "Add schedule" appends a new schedule row. Clicking "Remove" removes it. Cannot submit with 0 schedules.
- **ScheduleManager:** Each row has day selector, start time, end time. Time fields validate format.
- **useClasses:** With MSW mock returning 2 classes, the hook returns an array of length 2.
- **useCreateClass:** On success, `queryClient.invalidateQueries(['classes'])` is called.

### E2E (Playwright)

- **Teacher creates a class:** Login as teacher → `/classes/new` → fill name, martial art, add 2 schedules → submit → redirected to class detail → class appears in `/classes` list.
- **Ownership guard:** Login as Teacher B → attempt to navigate to Teacher A's class URL → redirected or shown 403 error state.
- **Student cannot access:** Login as student → navigate to `/classes/new` → redirected away.
