# v0.2 — Authentication

## Main Goal

Users can register, log in, view their profile, update it, and delete their account. The backend issues opaque access tokens (no JWTs, no PII in tokens). The frontend stores the token in memory only. After this ticket the app is a working, secure auth shell.

---

## Scope

### Backend

**Migrations (new)**

- No new migrations — `users`, `student_profiles`, `teacher_profiles` were created in v0.1.

**Models**

- `User` model with Lucid:
  - Encrypted fields: `last_name`, `email`, `birth_date` using `@adonisjs/encryption`
  - `password_hash` hashed with Argon2 via `@adonisjs/hash`
  - Soft delete via `deleted_at`
  - Relationships: `hasOne` StudentProfile, `hasOne` TeacherProfile
- `StudentProfile` and `TeacherProfile` models with encrypted fields

**Auth (AdonisJS Auth — API tokens)**

- Opaque access token driver (tokens stored hashed in DB, no JWT)
- Token TTL from `TOKEN_EXPIRY` env var (default 7d)
- Token revocation on logout (server-side)

**VineJS validators**

- `RegisterValidator`: email (valid format), password (min 8 chars), first_name, profile_type (`teacher` | `student`)
- `UpdateProfileValidator`: optional fields only, no email change without re-verify (defer to v0.7 scope — for now email is immutable after register)

**Controllers**

`AuthController`:

- `POST /api/v1/auth/register` — hash password, create user + profile row, issue token, return `{ token, user: { id, first_name, profile_type } }`
- `POST /api/v1/auth/login` — verify credentials, issue token, return same shape
- `DELETE /api/v1/auth/logout` — revoke token (auth middleware required), return `204`
- `GET /api/v1/auth/me` — return current user's own data (P2 fields visible to self: email, full name, birth_date)
- `PUT /api/v1/auth/me` — update profile fields (first_name, last_name, avatar_url, birth_date, weight_kg, height_cm for students)
- `DELETE /api/v1/auth/me` — anonymize account: set `first_name = 'Deleted'`, `last_name = encrypted('User')`, `email = encrypted(hash(original_email))`, `password_hash = ''`, `deleted_at = now()`. Delete student_profile / teacher_profile rows. Return `204`.

**Middleware**

- `@adonisjs/limiter` applied to auth routes: 5 attempts/min per IP for `register` and `login`
- Auth middleware (`auth.use('api')`) on all `/api/v1/auth/*` routes except `register` and `login`

**Input sanitization middleware**

- Strip HTML tags and script content from all string inputs before they reach the controller. Apply globally to all API routes.

**Audit logging**

- Log `login`, `logout`, `profile_update`, `account_delete` events to `audit_logs` via `AuditLogService.log(user, action, resource)`

**Serialization**

- `GET /api/v1/auth/me` exposes P2 fields (email, full name) to the authenticated user only
- No endpoint ever returns `password_hash`

### Frontend

**Auth domain structure**

```
src/domains/auth/
  schemas/
    login.schema.ts       # Zod: email + password
    register.schema.ts    # Zod: email, password, first_name, profile_type
  services/
    auth.service.ts       # register(), login(), logout(), getMe(), updateMe(), deleteMe()
  hooks/
    useLogin.ts           # useMutation wrapping auth.service.login
    useRegister.ts        # useMutation wrapping auth.service.register
    useMe.ts              # useQuery wrapping auth.service.getMe
  types/
    auth.types.ts         # User, AuthResponse types (inferred from Zod schemas)
  components/
    LoginForm.tsx         # RHF + Zod resolver, calls useLogin
    RegisterForm.tsx      # RHF + Zod resolver, profile_type selector, calls useRegister
    ConsentCheckbox.tsx   # Reusable consent acknowledgment checkbox
```

**Routes**

- `/login` — renders `LoginForm`, redirects to `/dashboard` if already authenticated
- `/register` — renders `RegisterForm`, redirects to `/dashboard` if already authenticated
- `_authenticated.tsx` layout — reads `useMe`, redirects to `/login` if unauthenticated

**Token lifecycle**

- On successful login/register: call `setToken(token)` from `api-client.ts` (in-memory, never localStorage)
- On logout or 401: call `clearToken()`, navigate to `/login`
- On page refresh: token is lost (expected — user must re-login). Token is NOT persisted across refreshes in this version.

**shared/hooks/useAuth.ts**

- Exposes `{ user, isLoading, isAuthenticated, login, logout, register }`

## Acceptance Criteria

- [ ] `POST /api/v1/auth/register` with valid data → `201` with `{ token, user: { id, first_name, profile_type } }`. `token` is an opaque string (not a JWT — does not decode to a JSON payload with PII).
- [ ] `POST /api/v1/auth/register` with duplicate email → `422` with a validation error message. The error does not reveal whether the email exists (generic "invalid credentials" style — privacy by design).
- [ ] `POST /api/v1/auth/login` with wrong password → `401`. After 5 attempts in 1 minute from same IP → `429`.
- [ ] `GET /api/v1/auth/me` without token → `401`.
- [ ] `GET /api/v1/auth/me` with valid token → `200` with full user data (email, full name visible to self).
- [ ] `DELETE /api/v1/auth/logout` revokes the token. Using the same token again → `401`.
- [ ] `DELETE /api/v1/auth/me` → `204`. The DB row has `deleted_at` set, `email` and `last_name` columns store ciphertext of anonymized values. `student_profile` / `teacher_profile` row is deleted.
- [ ] `password_hash` is never present in any API response.
- [ ] Registering as `teacher` creates a `teacher_profiles` row. Registering as `student` creates a `student_profiles` row.
- [ ] Frontend `/login` page submits credentials, receives token, stores in memory, and redirects to `/dashboard`.
- [ ] Visiting any `_authenticated` route without a token redirects to `/login`.
- [ ] Logging out clears the in-memory token and redirects to `/login`.

## Tests

### Backend (Japa)

- **Register — happy path:** `POST /register` with valid payload → `201`, token present, user row in DB, password not stored in plaintext.
- **Register — duplicate email:** Same email twice → second request returns `422` (or `409`).
- **Register — validation:** Missing `first_name`, invalid email, short password → `422` with field-level errors.
- **Login — wrong password:** → `401`.
- **Login — rate limit:** 6 rapid attempts from same IP → 6th returns `429`.
- **Token opacity:** The issued token does not split by `.` into three base64 parts (i.e., it is NOT a JWT).
- **Auth guard:** `GET /api/v1/auth/me` without token → `401`. With expired/revoked token → `401`.
- **Logout revocation:** Issue token → logout → use same token → `401`.
- **me — data isolation:** User A's token cannot retrieve User B's data via `GET /me` (obviously — but assert the endpoint always returns the authenticated user's own data).
- **Account deletion:** After `DELETE /me`, the DB row has `deleted_at` set and `email` column does not store the original email plaintext. The associated profile row is gone.
- **Anonymization — no PII:** After deletion, querying the raw DB row for the user shows no recognizable first name, last name, or email.
- **Audit log:** After login, a row exists in `audit_logs` with `action = 'login'` and the correct `user_id`.

### Frontend (Vitest)

- **LoginForm — validation:** Submit with empty email → shows "Email is required" error. Submit with invalid email format → shows format error. Does not call the API.
- **LoginForm — success:** Mock API returns `{ token, user }` → `setToken` is called with the token, user is redirected.
- **RegisterForm — profile_type:** Selecting `teacher` vs `student` changes which profile fields are shown.
- **useLogin — error handling:** Mock API returns `401` → mutation error state is set, no redirect.
- **ConsentCheckbox:** Is unchecked by default, form cannot submit while unchecked.

### E2E (Playwright)

- **Full auth flow:** Navigate to `/register` → fill form → submit → land on `/dashboard` → navigate to `/login` (should redirect to dashboard since authenticated) → refresh page → land on `/login` (token lost on refresh, expected) → log in again → log out → redirected to `/login`.
