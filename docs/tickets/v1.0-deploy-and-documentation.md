## Scope

### CI/CD — GitHub Actions

**`.github/workflows/ci.yml`** — runs on every push to `main` and on PRs:

```yaml
jobs:
  backend-test:
    - Checkout
    - Setup Node 20
    - Install deps (npm ci)
    - Start PostgreSQL service container (postgres:16-alpine)
    - Run migrations (node ace migration:run)
    - Run Japa tests (node ace test)

  frontend-test:
    - Checkout
    - Setup Node 20
    - Install deps (npm ci)
    - Run Vitest (npm run test)
    - Run build check (npm run build)

  e2e:
    - Checkout
    - Setup Node 20
    - Start full stack via docker-compose
    - Wait for health check
    - Run Playwright tests (npm run test:e2e)
    - Upload Playwright report as artifact
```

**No separate deploy workflow needed** — Render auto-deploys on push to `main` via its native Git integration. Each service (backend, frontend) has its own auto-deploy trigger based on root directory and build filters.

### Production Deployment — Render.com (Monorepo)

**Platform:** Render.com using **monorepo support** (see [Render Monorepo Docs](https://render.com/docs/monorepo-support)).

Both services live in the same Git repo. Each Render service sets a **root directory** so that builds, start commands, and auto-deploy triggers are scoped to only the relevant subdirectory. Build filters prevent unnecessary redeployments when changes are outside a service's directory.

**Blueprint — `render.yaml`** (at repo root):

```yaml
services:
  # ── Backend (Web Service) ──────────────────────────────
  - type: web
    name: fight-club-api
    runtime: node
    plan: free # or starter
    rootDir: backend
    buildCommand: npm ci && node ace.js migration:run
    startCommand: node ace.js serve
    buildFilter:
      paths:
        - backend/**
    envVars:
      - key: NODE_ENV
        value: production
      - key: HOST
        value: 0.0.0.0
      - key: PORT
        value: 3333
      - key: APP_KEY
        sync: false # set manually in Render dashboard
      - key: DATABASE_URL
        sync: false # Supabase connection string
      - key: REDIS_HOST
        sync: false
      - key: REDIS_PORT
        sync: false
      - key: REDIS_PASSWORD
        sync: false
      - key: VITE_APP_URL
        sync: false # frontend production URL
      - key: ANTHROPIC_API_KEY
        sync: false

  # ── Frontend (Static Site) ────────────────────────────
  - type: web
    name: fight-club-app
    runtime: static
    rootDir: frontend
    buildCommand: npm ci && npm run build
    staticPublishPath: dist
    buildFilter:
      paths:
        - frontend/**
    routes:
      - type: rewrite
        source: /*
        destination: /index.html # SPA fallback
    envVars:
      - key: VITE_API_URL
        sync: false # backend production URL
    headers:
      - path: /*
        name: X-Frame-Options
        value: DENY
      - path: /*
        name: X-Content-Type-Options
        value: nosniff
```

**Backend service:**

- Render Web Service with `rootDir: backend`.
- Build: `npm ci && node ace.js migration:run` — migrations run automatically on each deploy.
- Start: `node ace.js serve` (production mode, no HMR).
- Auto-deploy triggers only when files under `backend/` change.
- Environment variables set via Render dashboard (secrets never in repo).
- Domain: Render-assigned `*.onrender.com` subdomain (or custom domain).

**Frontend service:**

- Render Static Site with `rootDir: frontend`.
- Build: `npm ci && npm run build` → outputs to `dist/`.
- SPA fallback via `routes` rewrite rule (replaces Nginx).
- Auto-deploy triggers only when files under `frontend/` change.
- Domain: Render-assigned `*.onrender.com` subdomain (or custom domain).

**HTTPS:**

- Render provides managed TLS for all services (free, automatic).
- HTTP → HTTPS redirect enforced automatically by Render.
- `Strict-Transport-Security` header set by backend middleware (v0.8).
- Frontend static site headers configured in `render.yaml`.

### Production Seeding

**`node ace db:seed --files database/seeders/demo_seeder.ts`**

Demo seeder creates:

- 1 teacher user: `teacher@demo.com` / `Demo1234!`
- 1 student user: `student@demo.com` / `Demo1234!`
- 2 classes (one with belt system, one without)
- Schedules for each class
- 1 active invitation per class
- Student enrolled in both classes (with consent)
- 3 announcements across classes
- 2 feedback entries for the student
- 3 belt progress entries (for the belt-system class)
- 5 notifications (mix of read/unread)

Seeder is idempotent: running it twice does not duplicate data (check-before-insert pattern).

## Tests

### Infrastructure / Smoke Tests

- **Health check:** `GET /health` → `200` on the Render backend URL.
- **HTTPS redirect:** Render enforces HTTPS automatically; verify `curl -I http://<backend>.onrender.com/health` → `301` with `Location: https://...`
- **Security headers on prod:** `curl -I https://<backend>.onrender.com/health` returns `X-Content-Type-Options`, `X-Frame-Options`, `Strict-Transport-Security`.
- **CORS on prod:** Request from an unlisted origin → no `Access-Control-Allow-Origin` header in response.
- **Supabase connectivity:** Backend starts without DB errors; migrations run successfully during build step.
- **Demo credentials work:** Playwright smoke test:
  1. Navigate to live frontend URL
  2. Log in as `teacher@demo.com`
  3. Assert dashboard renders with at least 1 class card
  4. Log out → log in as `student@demo.com`
  5. Assert dashboard renders with at least 1 enrollment
- **No secrets in repo:** `git log --all -S 'ANTHROPIC_API_KEY' -- .` returns no commits (CI step).
- **Monorepo build filters:** Push a backend-only change → verify only the backend service redeploys. Push a frontend-only change → verify only the frontend service redeploys.

### CI Verification

- All jobs in `ci.yml` pass on a fresh clone with no cached state.
- Render auto-deploy triggers correctly on push to `main` (verify by checking `/health` response after deploy).
