# v0.9 — Testing

## Main Goal

Write a comprehensive test suite covering all critical paths: backend unit/integration/privacy tests with Japa and frontend component/hook/E2E tests with Vitest and Playwright. Every prior ticket's untested acceptance criteria is covered here. After this ticket the codebase has a reliable safety net for the final deploy.

#### Required Privacy Test Cases

**`data_isolation.spec.ts`**:

- Teacher A cannot see Teacher B's classes
- Teacher A cannot see Teacher B's students
- Student A cannot see Student B's feedback
- Student A cannot see Student B's belt history
- Student A cannot see Student B's enrollments
- Teacher cannot see students of classes they don't own
- Student enrolled in Class A cannot see announcements of Class B

**`encryption.spec.ts`**:

- `users.email` stored as ciphertext (raw DB value ≠ plaintext email)
- `users.last_name` stored as ciphertext
- `users.birth_date` stored as ciphertext
- `student_profiles.weight_kg` stored as ciphertext
- `student_profiles.height_cm` stored as ciphertext
- `feedback.content` stored as ciphertext
- All encrypted fields decrypt correctly on API read

**`anonymization.spec.ts`**:

- After `DELETE /auth/me`: `users.first_name` is "Deleted"
- After deletion: raw `email` column does not contain the original email
- After deletion: `student_profile` row is gone
- After deletion: enrollments are hard-deleted
- After deletion: feedback received is hard-deleted
- After deletion: audit logs retained (not deleted), `user_id` is anonymized/nulled
- Deleted user cannot log in

**`pii_in_api_responses.spec.ts`**:

- `GET /classes/:id/students` does not contain `email`, `last_name`, `birth_date`, `weight_kg`, `height_cm`
- `GET /announcements` does not expose `author_id` as a raw field
- Auth tokens do not decode as JWTs (not three base64 parts separated by `.`)
- Invitation URLs contain only UUID token — no class name or PII

- Token stored in memory, never written to `localStorage`/`sessionStorage`

- Requests without token have no `Authorization` header

**`LoginForm.test.tsx`**:

- Empty email → "Email is required" error
- Invalid email format → format error
- Wrong password response (mock 401) → shows error message

**`ConsentDialog.test.tsx`**:

#### Critical E2E Scenarios (must all pass)

1. **Full teacher flow:** Register → create class with 2 schedules → generate invite → copy invite URL
2. **Full student join flow:** Open invite URL → accept consent → register → join class → see class in dashboard
3. **Announcement flow:** Teacher posts announcement → student sees it → teacher deletes it → gone from student view
4. **Feedback loop:** Teacher sends feedback → AI suggestion used → student receives notification → student sees feedback
5. **Belt progression:** Teacher awards belt → student sees it in timeline → notification appears
6. **Privacy export:** Student exports data → JSON downloaded → contains expected fields
7. **Account deletion:** Student deletes account → cannot log back in → teacher's class still exists → student no longer appears in teacher's student list
8. **Data isolation (E2E):** Teacher B logs in → cannot access Teacher A's class URL → sees 403 or redirect
9. **Rate limiting:** 6 rapid login attempts → 6th shows rate limit error in UI
10. **Mobile responsive:** Core flows (login, view classes, view feedback) work at 375px width

## Acceptance Criteria

- [ ] `npm run test` (backend) passes all Japa tests with no failures.
- [ ] `npm run test` (frontend) passes all Vitest tests with no failures.
- [ ] `npm run test:e2e` passes all 10 critical Playwright E2E scenarios.
- [ ] Backend test coverage: ≥ 80% line coverage on services, controllers, and policies.
- [ ] All 5 privacy test files pass: data isolation, encryption, anonymization, PII-in-responses.
- [ ] Factories can generate all model types without errors.
- [ ] E2E tests run against a real backend instance (not mocked) seeded with test data.
- [ ] No test uses `any` TypeScript type to bypass type checking.
- [ ] Tests clean up after themselves (use transactions or `RefreshDatabase` trait — no test leaves data that affects another test).
